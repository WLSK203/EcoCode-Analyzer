## 1. Executive Summary

EcoCode Analyzer is an innovative static code analysis tool designed to measure and reduce the environmental impact of software by analyzing source code complexity and calculating carbon emissions. In an era where data centers consume approximately 1% of global electricity, this project addresses the critical gap between software development and environmental sustainability.

The tool analyzes source code using Abstract Syntax Tree (AST) parsing, detects algorithmic complexity using Big O notation, and converts computational metrics into carbon emissions using scientific formulas. It provides developers with actionable optimization suggestions and real-world environmental impact equivalents.

**Key Achievements:**
- Successfully implemented complexity detection for 8 complexity classes (O(1) to O(n!))
- Developed carbon calculation engine using research-backed methodologies
- Created optimization suggestion system with priority ranking
- Built interactive CLI with color-coded environmental impact reports
- Achieved 15+ Java classes with clean architectural design
- Implemented support for 4 programming languages

This project aligns with UN Sustainable Development Goals (SDG 13: Climate Action) and represents a novel intersection of computer science and environmental science.

## 2. Introduction

### 2.1 Background

Software development has traditionally focused on functionality, performance, and security. However, the environmental cost of running code has been largely ignored. With the exponential growth of cloud computing, mobile applications, and IoT devices, software now consumes massive amounts of energy.

The problem is amplified by inefficient algorithms. A function with O(n²) complexity instead of O(n log n) doesn't just run slower—it consumes exponentially more CPU cycles, electricity, and ultimately generates more carbon emissions.

### 2.2 Motivation

The motivation for this project stems from three key observations:

1. **Climate Crisis:** Technology is both a contributor to and potential solution for climate change
2. **Developer Awareness Gap:** Most programmers have no feedback on the environmental cost of their code
3. **Measurable Impact:** Algorithmic efficiency directly translates to energy consumption

By creating a tool that makes the invisible carbon cost visible, we can influence better coding practices at scale.

### 2.3 Project Scope

This project implements a Minimum Viable Product (MVP) that:
- Analyzes Java code with full AST parsing support
- Provides basic analysis for Python, JavaScript, and C/C++
- Calculates carbon emissions using scientific formulas
- Generates optimization suggestions with savings estimates
- Delivers results through an interactive command-line interface

## 3. Problem Statement

### 3.1 The Environmental Challenge

**Current State:**
- Data centers consume 200 TWh of electricity annually (1% of global consumption)
- Inefficient software can increase server load by 50-100%
- A single poorly optimized algorithm running in production can emit kilograms of CO₂
- Developers receive no feedback on environmental impact during development

**The Missing Link:**
While we have tools for code quality (SonarQube), performance profiling, and security scanning, there exists no tool to measure and report the environmental cost of code.

### 3.2 Core Problem Questions

1. How can we automatically analyze source code for computational efficiency?
2. How do we translate algorithmic complexity into quantifiable carbon emissions?
3. How can we make this information actionable for developers?
4. What optimizations have the highest environmental ROI?

### 3.3 Target Audience

- **Individual Developers:** Programmers wanting to write environmentally conscious code
- **Tech Companies:** Organizations pursuing sustainability goals
- **Educational Institutions:** CS professors teaching algorithmic efficiency
- **Open Source Projects:** Communities optimizing for sustainability

└───────┬────────┘ └─────────────┘ └─────────────────┘
        │
┌───────▼────────┐
│  Code Parser   │

└────────────────┘
        │
┌───────▼────────┐

### 7.2 Component Descriptions

#### User Interface Layer
- **Main.java**: Entry point, command-line argument parsing, interactive menu
- **ReportFormatter.java**: ANSI-colored console output formatting

#### Application Layer
- **CodeAnalyzer.java**: Main orchestrator coordinating all components

#### Core Business Logic
- **ComplexityDetector.java**: AST traversal and complexity pattern matching
- **CarbonCalculator.java**: Energy and emission calculations
- **OptimizationEngine.java**: Pattern detection and suggestion generation

#### Data Layer
- **Models Package**: 7 model classes representing domain entities

### 7.3 Data Flow

1. **Input:** User provides file path via CLI
2. **Parsing:** CodeAnalyzer reads file → JavaParser builds AST
3. **Analysis:** ComplexityDetector traverses AST → detects patterns → returns ComplexityResult
4. **Calculation:** CarbonCalculator takes complexity → estimates operations → calculates carbon
5. **Optimization:** OptimizationEngine analyzes code patterns → generates suggestions
6. **Reporting:** ReportFormatter compiles results → formats output → displays to user

### 8.2 Design Patterns Used

#### 1. Facade Pattern (CodeAnalyzer)
**Purpose:** Simplifies complex subsystem interaction  
**Implementation:** CodeAnalyzer provides simple `analyzeFile()` method hiding complexity detector, calculator, and optimizer coordination

#### 2. Visitor Pattern (AST Traversal)
**Purpose:** Separate algorithms from object structure  
**Implementation:** LoopAnalysisVisitor, LoopPatternVisitor traverse AST nodes

#### 3. Builder Pattern (Report Construction)
**Purpose:** Construct complex objects step-by-step  
**Implementation:** CarbonReport built incrementally with function analyses and suggestions

#### 4. Strategy Pattern (Parsers)
**Purpose:** Interchangeable algorithms  
**Implementation:** Different parsing strategies for Java, Python, JS, C++

#### 5. Enum Singleton (Complexity)
**Purpose:** Type-safe, guaranteed single instances  
**Implementation:** Complexity enum ensures one instance per complexity class

#### 6. Repository Pattern (Database - Planned)
**Purpose:** Abstract data access  
**Implementation:** AnalysisRepository for SQLite history (stubbed)

### 8.3 Key Algorithms

#### Complexity Detection Algorithm

```
function detectComplexity(method):
    1. Initialize loop nesting counter
    2. Traverse AST using Visitor pattern:
        a. For each loop:
            - Increment nesting level
            - Check for nested patterns
        b. For each recursive call:
            - Count invocations
            - Detect multiple recursion (Fibonacci)
        c. For each variable pattern:
            - Detect divide/conquer (i /= 2)
            - Detect multiply patterns (i *= 2)
    3. Match against known algorithms:
        - Binary search: left/right pointers
        - Sorting: compare/swap patterns
    4. Apply heuristics:
        - Single loop → O(n)
        - Nested loops → O(n²)
        - Divide recursion → O(log n)
        - Multiple recursion → O(2^n)
    5. Calculate confidence score
    6. Return ComplexityResult
```

#### Carbon Calculation Algorithm

```
function calculateCarbon(complexity, inputSize):
    1. Estimate operations = complexity.estimateOperations(inputSize)
    2. Calculate CPU cycles:
        cycles = operations × baseInstructions / IPC
    3. Calculate time:
        time_seconds = cycles / (clockSpeed × 10⁹)
    4. Calculate energy:
        energy_Wh = cpuPower × (time_seconds / 3600)
    5. Calculate carbon:
        carbon_g = energy_Wh × carbonIntensity / 1000
    6. Return carbon emissions in gCO₂
```

## 10. Detailed Implementation

### 10.1 Models Package

#### Complexity.java (84 lines)
Enum defining Big O complexity classes with mathematical operations estimation.

**Key Features:**
- 8 complexity classes from O(1) to O(n!)
- `estimateOperations(int n)` method for each class
- Severity levels for prioritization
- Factorial helper function for O(n!)

**Example:**
```java
O_N_SQUARED.estimateOperations(1000) → 1,000,000 operations
```

#### ComplexityResult.java (82 lines)
Stores the result of complexity analysis for a code block.

**Fields:**
- `functionName`: Name of analyzed function
- `timeComplexity, spaceComplexity`: Detected complexities
- `confidenceScore`: 0.0-1.0 detection confidence
- `detectionMethod`: How complexity was determined
- `details`: List of detection evidence

#### CarbonReport.java (149 lines)
Comprehensive carbon emission report for a file or project.

**Key Methods:**
- `calculateTotalEmissions()`: Sums all function emissions
- `calculateTotalTime()`: Aggregate execution time
- `findWorstFunction()`: Identify highest emitter
- `getRating()`: 1-5 star environmental rating

#### FunctionAnalysis.java (86 lines)
Per-function carbon metrics and analysis results.

**Metrics:**
- Carbon emissions (gCO₂)
- Estimated execution time (ms)
- Energy consumption (Wh)
- Input size and LOC

#### EnvironmentalMetrics.java (109 lines)
Converts carbon emissions to real-world equivalents.

**Conversions:**

#### OptimizationSuggestion.java (164 lines)

**Priority Levels:**

**Suggestion Types:**

#### CodeFile.java (104 lines)

#### ComplexityDetector.java (282 lines)
AST-based complexity detection engine.

**Detection Strategies:**

1. **Loop Nesting:**
```java
- Single loop → O(n)
- Nested 2 levels → O(n²)
- Nested 3 levels → O(n³)
```

2. **Recursion:**
```java
- Single recursive call → O(n)
- Multiple calls → O(2^n)
- Divide-and-conquer → O(n log n)
```

3. **Pattern Matching:**
```java
- Binary search (left/right) → O(log n)
- Loop division (i/=2) → O(log n)
- Sort patterns → O(n log n)
```

**AST Visitors:**
- `LoopAnalysisVisitor`: Counts loop nesting
- `LoopPatternVisitor`: Detects logarithmic patterns

#### CarbonCalculator.java (120 lines)
Energy and carbon emission calculation engine.

**Configuration:**
```java
CPU_POWER_WATTS = 65.0
CARBON_INTENSITY = 475.0  // gCO₂/kWh
CPU_CLOCK_SPEED_GHZ = 3.5
INSTRUCTIONS_PER_CYCLE = 2.0
BASE_INSTRUCTIONS_PER_OP = 100
```

**Formula Implementation:**
```java
double operations = complexity.estimateOperations(inputSize);
double cycles = (operations * BASE_INSTRUCTIONS) / IPC;
double timeSeconds = cycles / (CLOCK_SPEED * 1e9);
double energyWh = CPU_POWER * (timeSeconds / 3600);
double carbonGrams = energyWh * CARBON_INTENSITY / 1000;
```

#### OptimizationEngine.java (230 lines)
Pattern detection and suggestion generation.

**Detection Rules:**

1. **Nested Loops (O(n²)):**
```
Suggestion: Use HashMap for O(1) lookup
Savings: ~90% for large n
```

2. **ArrayList.contains() in loop:**
```
Suggestion: Replace with HashSet
Savings: O(n²) → O(n)
```

3. **Bubble/Selection Sort:**
```
Suggestion: Use QuickSort/MergeSort/Arrays.sort()
Savings: O(n²) → O(n log n)
```

4. **Recursive Fibonacci:**
```
Suggestion: Use memoization or dynamic programming
Savings: O(2^n) → O(n)
```

## 13. Challenges and Solutions

### Challenge 1: Complexity Detection Accuracy

**Problem:** Determining Big O complexity from code is theoretically undecidable (Halting Problem).

**Solution:** 
- Implemented heuristic-based approach
- Pattern matching for common algorithms
- Confidence scoring system
- Focus on detectable patterns (loops, recursion)
- Accept limitations: won't catch every edge case

### Challenge 2: Multi-Language Support

**Problem:** Each language has different syntax and parsing requirements.

**Solution:**
- Full implementation for Java using JavaParser
- Stub implementations for other languages
- Modular parser architecture for future expansion
- ANTLR grammar integration planned

### Challenge 3: Realistic Carbon Calculations

**Problem:** CPU behavior varies widely; how to get realistic estimates?

**Solution:**
- Researched IEEE papers on software energy consumption
- Used industry-standard assumptions (65W CPU, 475g/kWh carbon intensity)
- Focused on relative comparisons rather than absolute accuracy
- Documented all assumptions transparently

### Challenge 4: Making Data Relatable

**Problem:** "0.012 gCO₂" doesn't mean much to most people.

**Solution:**
- Convert to real-world equivalents
- Driving distance most intuitive
- Tree absorption for environmental context
- Visual ratings (stars) for quick assessment

### Challenge 5: Performance with Large Files

**Problem:** AST parsing can be slow for large codebases.

**Solution:**
- Efficient visitor patterns
- Minimal AST traversals
- Focus analysis on functions, not entire file
- Project analysis runs in parallel (planned)
